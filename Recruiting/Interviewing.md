### **How to Answer Questions**
---

**Framework: STARRS**
- **S**ituation -- Set the context for your story.
- **T**ask -- What was the task required of you?
- **A**ctivity -- What you did. Be specific. Use "I" not "we."
- **R**esult -- How well did the situation play out? Quantify.
- **R**eflect -- What did you do well? What didn't go as well? What would you have done differently?
- **S**trengthen -- How would you approach a similar situation in the future?

**Amazon Leadership Principles (weave into every answer):**
1. **Customer Obsession** -- Start with the customer and work backwards.
2. **Ownership** -- Think long term. Never say "that's not my job."
3. **Invent and Simplify** -- Expect and require innovation. Find ways to simplify.
4. **Dive Deep** -- Operate at all levels. No task is beneath you. Audit and verify.
5. **Bias for Action** -- Speed matters. Many decisions are reversible.
6. **Insist on the Highest Standards** -- Continually raise the bar.
7. **Earn Trust** -- Listen attentively, speak candidly, treat others respectfully.

**Your Competitive Advantages (use these as themes across answers):**
- You founded and lead a real client-facing software consultancy -- most interns have never managed stakeholders or shipped to real users
- You have a production system actively saving $50/month (AWS Cost Optimizer) -- not just a class project
- Full-stack depth across Python, REST APIs, JUnit testing, and deep S3 API knowledge
- You think in systems (risk scoring, rollback safety, security layers) -- not just features
- You've led through ambiguity repeatedly -- changing client requirements, teams needing direction, new technologies with no one to ask
- You're already integrating AI into your engineering workflow for adversarial testing and edge case generation

**Interview Format (per recruiter):**
- 4 behavioral questions total
- 2 soft behavioral (leadership, teamwork, communication)
- 2 technical behavioral (debugging, architecture decisions, technical problem-solving)

### **2. Why Amazon?**

- **Answer:**

	- Honestly, I built my way into this answer before I even applied. I spent months building an AWS Cost Optimizer -- a desktop app that talks directly to the S3 API to scan buckets, score optimization opportunities by risk, and execute storage class transitions with full rollback. I chose S3 as the target because it sits at this intersection of deceptive simplicity and genuine depth: the API surface looks straightforward, but once you dig in you hit pagination semantics on `list_objects_v2`, the fact that "changing a storage class" is actually a `copy_object` call back to the same key, lifecycle rule merge conflicts when multiple tools touch `put_bucket_lifecycle_configuration` -- real operational complexity that customers deal with every day. Building that system taught me that S3 is not just a storage service, it is infrastructure that the entire cloud runs on, and the team building it has to obsess over durability, cost, and safety at a scale that no other team gets to touch. That is what I want to be part of. Customer Obsession at the S3 level means that when a customer stores an object, they trust it will be there -- eleven nines of durability is not a marketing number, it is an engineering commitment enforced at every layer. I want to work on systems where "good enough" does not exist.

	The other piece is how Amazon thinks. The Leadership Principles are not wall art -- they are decision-making tools. When I was designing my optimizer's execution pipeline, I had to make a call: do I let users batch-execute all recommendations at once, or do I force progressive execution modes -- dry-run, then single-object, then filtered batch? The "fast" answer was batch-execute-all. The right answer was progressive modes, because Customer Obsession means you do not let a user accidentally transition 10,000 production objects to Glacier Deep Archive with one click. That is the same kind of thinking I see in how S3 designs its APIs -- deliberate, safe, auditable. I want to bring that mindset to the team that invented it.

### **9. Walk me through a technical architecture decision you made and why.**

- *What they're looking for:* Systems thinking, tradeoff analysis, justifying decisions with S3 API depth, safety-first mutation design.
- *Amazon LPs:* Dive Deep (S3 API semantics and copy-object mechanics), Customer Obsession (safe mutations with rollback), Are Right A Lot (design tradeoffs on execution modes), Ownership (end-to-end scan-to-rollback pipeline), Invent and Simplify (progressive execution modes).

- **Answer (STARRS) -- AWS Cost Optimizer Scan-Score-Execute-Rollback Pipeline:**

	- **Situation:** I was building my AWS Cost Optimizer -- a tool that scans S3 buckets, identifies cost optimization opportunities, and executes storage class transitions. The core architecture decision was not "what framework" -- it was how to design a four-phase pipeline (scan, score, execute, rollback) that could safely mutate production S3 data without any irreversible mistakes. Every phase exists because S3's API semantics force specific design constraints that are not obvious until you dive deep into them.

	- **Task:** I needed to architect a pipeline that could enumerate millions of objects across buckets, score each one for optimization risk, execute transitions safely, and provide full rollback -- all while respecting S3's API quirks: there is no native "change storage class" API, lifecycle rules are bucket-global and can conflict, and multipart uploads silently accumulate cost if abandoned.

	- **Activity:** I made four interconnected architecture decisions, each driven by a specific S3 API constraint.

		**Phase 1 -- Scan:** S3's `list_objects_v2` returns at most 1,000 keys per call, so I built the scanner around boto3 paginators that handle continuation tokens automatically. For each object, I capture `StorageClass`, `LastModified`, `Size`, and `ContentType` and persist them to a local SQLite database as a pre-mutation snapshot. I also call `list_multipart_uploads` per bucket to find abandoned multipart uploads -- these are invisible cost sinks that do not show up in `list_objects_v2` at all. Every scan writes a timestamped snapshot so we have a complete record of bucket state before any mutation happens.

		**Phase 2 -- Score:** Each object gets a 0-100 risk score based on five weighted factors: reversibility of the target storage class, data loss risk, age confidence (is it actually cold -- I use a 90-day threshold), size impact on cost, and access recency confidence. The scoring engine assigns different weights and produces a recommendation with full transparency -- the API response includes every individual factor score so the user sees exactly why an object scored the way it did. Objects below the safe-automation threshold get flagged for human review instead of auto-execution.

		**Phase 3 -- Execute:** This is where the deepest S3 API knowledge mattered. S3 has no `change_storage_class` API. To transition an object from STANDARD ($0.023/GB/month) to GLACIER_IR ($0.004/GB/month), you must call `copy_object` with the source and destination as the same key, setting `StorageClass` to the target class. Critically, you must pass `MetadataDirective="COPY"` and `TaggingDirective="COPY"` or you silently lose all user-defined metadata and tags on the object -- a data loss bug that would be invisible until someone tries to read those tags downstream. For abandoned multipart uploads, I call `abort_multipart_upload` with the specific `UploadId`. For lifecycle-based optimizations, I call `put_bucket_lifecycle_configuration`, but I implemented a merge strategy instead of a replace -- I read the existing lifecycle rules first and merge my new rule in, because a naive PUT would overwrite any lifecycle rules the customer already had configured, potentially breaking their existing retention policies.

		The key design tradeoff was execution modes. I could have built a single "execute all" button, but that is dangerous. Instead, I built three progressive execution modes: dry-run (shows what would happen, mutates nothing), single-object execution (test one transition, verify it worked), and filtered batch execution (apply to a scoped subset). This forces users to build confidence incrementally before touching their entire bucket. I also capture the original `StorageClass` of every object before mutation so rollback is possible.

		**Phase 4 -- Rollback:** Because every transition is a `copy_object` call, rollback is another `copy_object` call back to the original storage class, using the pre-mutation snapshot stored in SQLite. This only works because I captured state before mutation -- without that snapshot, rollback would be impossible since S3 does not maintain a storage class history.

	- **Result:** The pipeline runs on my own AWS account and saves roughly $50/month by transitioning cold objects (untouched for 90+ days) from STANDARD at $0.023/GB/month to GLACIER_IR at $0.004/GB/month -- an 82% reduction in per-GB storage cost. The system has 245+ tests with 80%+ coverage, including edge cases for paginator exhaustion, metadata preservation through copy-object, and lifecycle rule merge conflicts. Zero objects have been lost or had metadata silently stripped, because the `MetadataDirective` and `TaggingDirective` flags are enforced at the architecture level, not left to the caller.

	- **Reflect:** The biggest lesson was that S3's API simplicity is deceptive. The API surface is small, but the semantic implications of each call are deep -- `copy_object` without the right directives silently destroys metadata, `put_bucket_lifecycle_configuration` silently overwrites existing rules, `list_objects_v2` does not surface multipart uploads. Every phase of my pipeline exists because I dove deep into an API behavior that would have caused silent data corruption if I had taken the naive path. On the S3 team, this is the exact kind of thinking I would bring: understanding API semantics from the customer's perspective, not just the spec.

	- **Strengthen:** If I were extending this system, I would add versioned-bucket awareness -- right now my rollback relies on `copy_object`, but for versioned buckets I could leverage `s3:GetObjectVersion` to restore exact previous versions instead of re-copying. I would also add cost projection modeling that accounts for retrieval fees on Glacier classes, so users can see net savings after factoring in occasional access costs. On the Amazon S3 team, I would apply this same principle of building safety into the architecture rather than bolting it on after -- making the safe path the default path, and forcing users to opt into risk rather than opt out of it.

It's denormalized in a way that can't support reassignment history or dynamic routing. I need to rebuild it." He appreciated the honesty but was clear that we couldn't slip the sprint deadline.

### **24. What is indexing and what is the tradeoff?**

- *What they're looking for:* Understanding of database fundamentals, performance thinking, ability to explain tradeoffs clearly.
- *Amazon LPs:* Execute with Excellence (deliver seamlessly -- performance matters at scale).

- **Answer (STARRS):**

	- **Situation:** In Tech Startup Club, when we built the web application for UW Tacoma using Spring Boot and PostgreSQL, one of the pages loaded a list of all resources filtered by category and sorted by date. Early on it was fine, but as we seeded more test data to simulate realistic usage, that page started lagging noticeably. The query was doing a full table scan every time -- PostgreSQL was reading every row in the table, checking if the category matched, and then sorting the results by date. With a few hundred rows it was instant. With a few thousand, it was sluggish.

	- **Task:** I needed to understand why the query was slow and fix it without overcomplicating our database setup.

	- **Activity:** An index is essentially a separate data structure -- usually a B-tree -- that the database maintains alongside your table. Instead of scanning every row to find matches, the database can look up the index and jump directly to the relevant rows, similar to looking something up in a book's index instead of reading every page. I added a composite index on the category and date columns, which meant PostgreSQL could both filter by category and return results in date order using a single index lookup instead of a full scan plus a sort.

		But the tradeoff is real. Every index you add has costs: **write performance** takes a hit because every INSERT, UPDATE, or DELETE now has to update both the table and the index. **Storage** increases because the index is a separate data structure sitting on disk. And **maintenance overhead** grows -- if you index everything, you end up with indexes the query planner doesn't even use, which waste space and slow down writes for no read benefit. The right approach is to index the columns you actually query against frequently and leave the rest alone. In our case, category and date were the two columns that appeared in WHERE and ORDER BY clauses on the most-used page, so the composite index was justified. I didn't index every column just because I could.

		I verified the improvement by running EXPLAIN ANALYZE on the query before and after -- the full table scan turned into an index scan, and the sort step disappeared entirely because the index already stored rows in date order.

	- **Result:** The page load time dropped significantly after the index. More importantly, I understood *why* it worked rather than just copying a Stack Overflow answer. When we later added more filtered views to the app, I was able to make informed decisions about which columns to index based on actual query patterns rather than guessing.

	- **Reflect:** The lesson is that indexing isn't free and it isn't magic -- it's a tradeoff between read speed and write cost. For a read-heavy application like ours where users browse and filter far more than they create or update, the tradeoff heavily favored indexing. But in a write-heavy system -- say, a logging pipeline or a real-time event stream -- over-indexing can become a bottleneck because every write has to update multiple indexes. The right answer is always "it depends on the access pattern."

	- **Strengthen:** I now think about access patterns before designing schemas. When I create a table, I ask: what queries will hit this table most often? Which columns will appear in WHERE, JOIN, and ORDER BY clauses? I index based on those answers, not by default. On the S3 team at Amazon, where metadata systems handle billions of objects with both high read and high write loads, getting indexing decisions right is critical -- you can't afford full table scans on an object metadata table, but you also can't afford to slow down every PUT with unnecessary indexes.

### **25. What is a difficult bug you've ran into and how did you fix it?**

- *What they're looking for:* Ability to trace data flow through a system, identify where assumptions break down, and fix the root cause rather than the symptom.
- *Amazon LPs:* Dive Deep (traced inflation through scanner-to-lifecycle data flow), Earn Trust (users making financial decisions need accurate numbers), Customer Obsession (wrong savings estimates lead to wrong business decisions), Are Right A Lot (validated fix with pricing math), Insist on Highest Standards (redesigned the return type rather than adding a patch).

- **Answer (STARRS):**

	- **Situation:** In my AWS Cost Optimizer, I was doing an adversarial code review of the lifecycle recommendation module when I noticed the savings estimates looked suspiciously high. One of my test buckets had 100GB total across multiple storage classes -- about 10GB in STANDARD, 40GB in GLACIER_IR, and 50GB in DEEP_ARCHIVE -- but the lifecycle recommendation was reporting $2.19/month in potential savings. I did the math by hand: at $0.019/GB/month savings (the delta between STANDARD at $0.023 and GLACIER_IR at $0.004), 10GB of STANDARD objects should yield roughly $0.19/month. The system was reporting over 10x that amount.

	- **Task:** I needed to find where the savings inflation was coming from and fix it so that users making real financial decisions based on these numbers could trust them.

	- **Activity:** I traced the data flow from scanner to savings estimate. The scanner's `_scan_objects()` method iterated over every object in the bucket using `list_objects_v2` pagination and accumulated `total_size_bytes` by summing every object's `Content-Length`. This total was then passed to `_check_lifecycle()`, which computed savings as `round((_STANDARD_PRICE - _GLACIER_IR_PRICE) * size_gb, 4)`. The problem was clear: `_check_lifecycle()` was applying the STANDARD-to-GLACIER_IR price differential to ALL objects in the bucket, including the 40GB already in GLACIER_IR and the 50GB in DEEP_ARCHIVE. Those objects would generate zero savings from a lifecycle transition -- they're already in cheaper classes -- but the formula was counting them as if they were all in STANDARD.

		The fix required changing the return type of `_scan_objects()`. It previously returned `list[Recommendation]` -- just the list of recommendations it generated. I changed it to `tuple[list[Recommendation], int, int]` -- a 3-tuple of (recommendations, total_size_bytes, standard_size_bytes). Inside the object iteration loop, I added a conditional accumulator: `if storage_class_raw == "STANDARD": standard_size_bytes += size_bytes`. Every object still contributes to `total_size_bytes` (used for other reporting), but only STANDARD objects contribute to `standard_size_bytes`. I then updated `_check_lifecycle()` to receive `standard_size_bytes` instead of `total_size_bytes` for its savings computation.

		The savings formula itself didn't change -- `round((_STANDARD_PRICE - _GLACIER_IR_PRICE) * size_gb, 4)` -- but now `size_gb` is derived from `standard_size_bytes` instead of `total_size_bytes`. The pricing constants are STANDARD = $0.023/GB/month and GLACIER_IR = $0.004/GB/month, so the per-GB savings is $0.019/month, but that $0.019 only applies to bytes currently in STANDARD.

	- **Result:** After the fix, the lifecycle savings estimate for my test bucket dropped from $2.19/month to $0.19/month -- the mathematically correct number. I verified by hand: 10GB STANDARD * $0.019/GB/month = $0.19/month. Every savings estimate in the system now reflects only the objects that would actually benefit from a transition. If I hadn't caught this, a user with a 10TB bucket where only 500GB is in STANDARD would have seen $190/month in projected savings instead of the real $9.50/month -- and when they executed the recommendation and their bill only dropped by $9.50, they'd lose trust in the tool entirely.

	- **Reflect:** The bug was subtle because the code was "correct" in isolation -- the formula was right, the scan was right, the lifecycle check was right. The error was in the data flowing between them: the scan counted everything, but the lifecycle computation assumed everything was STANDARD. It was a semantic mismatch at a function boundary, and those are the hardest bugs to catch because no single function looks wrong. The only way I found it was by computing the expected answer independently and comparing it against the system's output -- a technique I learned to apply routinely: never trust a number your system produces without a back-of-envelope sanity check.

	- **Strengthen:** I now write assertion-style sanity checks into any module that produces financial estimates. If computed savings exceed total storage cost, something is wrong. If savings per GB exceeds the most expensive storage class price, something is wrong. These invariant checks turn silent data bugs into loud failures. On Amazon's S3 team, where billing accuracy directly affects customer trust and revenue, I'd apply the same principle: every cost computation should have bounds checks that scream when the math doesn't add up.

### **17. How do you approach security when designing a system? Walk me through your thought process.**

- *What they're looking for:* Defense-in-depth thinking, threat modeling instincts, understanding that security is a design constraint not a bolt-on. Do you think about attack surfaces proactively?
- *Amazon LPs:* Obsess about Customers (protect their data and trust), Execute with Excellence (deliver seamlessly and securely), Lead Courageously (make hard tradeoffs for safety).

- **Answer (STARRS):**

	- **Situation:** When I was designing my AWS Cost Optimizer, the system needed to handle AWS IAM access keys -- credentials that, if leaked, give full access to a user's cloud infrastructure. This isn't like leaking a password that can be reset. A compromised IAM key can spin up thousands of EC2 instances for crypto mining, delete production databases, or exfiltrate data from S3 -- and the user gets the bill. So security wasn't a feature I could add later. It had to be the foundation the entire architecture was built on.

	- **Task:** I needed to design a credential handling system where AWS keys were never stored in plaintext, never transmitted over a network, and never accessible to any process that didn't strictly need them. At the same time, the app still needed to use those credentials to make S3 API calls, so I couldn't just lock them away -- I had to design a secure path from storage to usage.

	- **Activity:** I thought about security in layers, where each layer assumes the one above it has already been breached. **Layer 1 -- Credential isolation:** AWS credentials are stored in the OS-native keychain (macOS Keychain, Windows Credential Manager). The keychain encrypts at rest and ties access to the user's login session, so even if someone copies the application's data directory, they get nothing. Credentials are never written to a config file, environment file, or database. **Layer 2 -- Process isolation:** When the Tauri frontend spawns the Python sidecar backend, it injects the credentials as environment variables scoped to that child process only. They exist in memory for the duration of the process and disappear when it terminates. No credential ever touches disk in plaintext. **Layer 3 -- Network isolation:** Tauri's Content Security Policy restricts the WebView frontend to connecting only to `http://127.0.0.1:8000`. Even if someone injected malicious JavaScript into the frontend, it physically cannot make outbound network requests to exfiltrate data. **Layer 4 -- Execution guardrails:** Even with valid credentials, the system defaults to `dry_run` mode -- no mutations. Destructive operations like object deletion are blocked unless `ALLOW_DESTRUCTIVE_EXECUTION` is explicitly enabled AND the user's IAM permissions include `s3:DeleteObject`. Every mutation captures pre-change state for rollback. **Layer 5 -- Update security:** Desktop application updates are signed with minisign and verified before installation, preventing supply chain attacks through compromised update packages.

	- **Result:** The result is that a user's AWS credentials never leave their machine, are never stored in plaintext, and even if every other layer fails, the execution guardrails prevent unintended mutations. The system has been running on my own AWS account for months with real credentials and real money at stake.

	- **Reflect:** The tradeoff of this approach is engineering complexity. Storing credentials in a simple `.env` file would have taken five minutes instead of the days I spent integrating OS keychains across three platforms. But the alternative -- asking users to trust a config file with their AWS keys in plaintext -- would have made the tool irresponsible to distribute. In security engineering, the question isn't "is this the easiest way?" -- it's "what's the blast radius if this layer fails?"

	- **Strengthen:** My mental model now for any system that handles sensitive data is: assume each layer will be compromised and ask what the next layer prevents. At Amazon, where S3 stores some of the most critical data on the internet for millions of customers, this defense-in-depth approach is exactly how I'd think about access control and data protection -- every layer should independently limit what an attacker can do, so a single breach doesn't cascade into data exposure or loss.

	- **Situation:** Early in the AWS Cost Optimizer project, I had a working prototype where the Python backend could execute any S3 operation the user's IAM credentials allowed -- including deleting objects permanently. The feature worked correctly: if a recommendation said "delete this stale object," the executor would delete it. But I realized that "correct" and "safe" are not the same thing.

	- **Task:** I needed to design a guardrail system that prevented the tool from doing irreversible damage, even when the user explicitly asked it to. The risk was that a misconfigured scan could flag active production objects as "stale" -- our access pattern detection relies on `LastModified` timestamps, which don't capture read activity -- and the executor would permanently delete them.

	- **Activity:** I designed a three-part mitigation. First, I introduced **progressive execution modes**: `dry_run` (default, no mutations), `safe` (only low-risk, reversible actions like storage class transitions), `standard` (adds medium-risk actions), and `full` (everything including deletions). The default is dry-run, so a user has to consciously escalate. Second, I added a **destructive action flag** (`ALLOW_DESTRUCTIVE_EXECUTION`) that defaults to `false`. Even in `full` mode, deletions are blocked unless this flag is explicitly set. This means you need two deliberate actions to delete anything -- escalate the mode AND enable the flag. Third, I built **pre-change state capture**: before every mutation, the system records what the object looked like before (storage class, lifecycle rules, metadata) so that non-destructive actions can be rolled back. For deletions, I documented clearly that rollback is not possible, which reinforces the gravity of enabling it. The simpler option would have been a single "are you sure?" confirmation dialog. But confirmation dialogs are security theater -- people click through them. The progressive mode system forces the user to understand the risk level before they ever reach the dangerous operation.

	- **Result:** In months of running the tool on my own AWS account, the guardrails have prevented unintended deletions multiple times. I've run full scans that flagged objects I didn't want deleted, and because the default mode is `dry_run`, I caught them in the preview before any mutation happened. Every execution produces an audit trail showing exactly what was attempted, what was blocked, and why.

	- **Reflect:** The tradeoff was real -- implementing four execution modes, the destructive flag, and the pre-state capture system was significantly more work than a simple "execute all recommendations" button. It also adds friction to the user experience. But for a tool that modifies cloud infrastructure with real financial consequences, friction is a feature, not a bug. The principle I took away is: **the more irreversible the action, the more deliberate the path to reach it should be.**

	- **Strengthen:** This maps directly to how I think S3 approaches data safety -- you don't let a destructive operation go through on a single signal. There are multiple layers of protection (bucket policies, object versioning, MFA delete, lifecycle rules) and the system is designed so that failing safely is always the default. If I were working on S3's storage management features, I'd apply the same principle: make the safe path the default, and require escalating levels of confidence before allowing higher-risk operations.

### **19. How do you think about the tradeoff between security and user experience?**

- *What they're looking for:* Nuanced thinking about security as a spectrum, not binary. Understanding that over-securing can make a system unusable and under-securing can be catastrophic.
- *Amazon LPs:* Obsess about Customers (balance protection with usability), Execute with Excellence (deliver seamlessly), Lead Courageously (make hard tradeoffs).

- **Answer (STARRS):**

	- **Situation:** In the AWS Cost Optimizer, I had to decide how the application handled credential re-entry. The OS keychain stores credentials persistently, but what happens when a user wants to switch AWS accounts or update expired keys? The secure answer is to prompt for credentials every time the app launches. The usable answer is to remember them forever and never ask again.

	- **Task:** I needed to find a middle ground where credentials were persistent enough that the app didn't feel annoying, but the system still validated that stored credentials were actually working before using them.

	- **Activity:** I designed a three-tier approach. First, credentials persist in the OS keychain across app launches -- the user doesn't re-enter them every time. This is secure because the keychain is encrypted at rest and tied to the user's OS login session. Second, every time the app starts, the Rust backend polls the health endpoint, and the Python sidecar validates the stored credentials by making a lightweight S3 `list_buckets` call. If the credentials are expired or revoked, the health check fails and the UI shows an "offline" banner with a prompt to update credentials in Settings -- rather than silently failing or making API calls with bad keys. Third, when the user updates credentials in Settings, the system kills the existing sidecar, writes new credentials to the keychain, spawns a fresh sidecar with the new credentials injected, and re-polls health -- all within a 15-second timeout. The user sees a loading state, then either a green status or an error telling them exactly what went wrong. The tradeoff I explicitly chose NOT to make was adding a session timeout or requiring periodic re-authentication. For a desktop tool running locally with locally-stored credentials, session timeouts add friction without meaningfully improving security -- the threat model is "someone with access to your unlocked machine," and if they have that, a session timeout doesn't help.

	- **Result:** Users set up credentials once, the app validates them silently on every launch, and credential changes are handled gracefully without restarting the application. The experience feels seamless while the underlying system is doing real validation.

	- **Reflect:** The lesson was that good security UX is invisible when things are working and informative when they're not. The user shouldn't feel the security -- they should only see it when something actually requires their attention (expired keys, permission errors). Overloading users with security prompts when there's no actual threat trains them to click through warnings, which makes the real warnings less effective.

	- **Strengthen:** At Amazon, this tradeoff is central to S3's design philosophy -- access controls and data protection need to be strong enough to prevent unauthorized access but seamless enough that developers can build on S3 without fighting the security model. The best security is the kind the user never notices until it saves them. I'd bring this mindset to S3: validate aggressively in the background, surface friction only when risk is genuinely elevated, and make error states clear and actionable rather than cryptic.

### **20. Walk me through your Cloud Optimizer project.**

- **Answer:**

	- AWS Cost Optimizer is a full-stack desktop application I built to help engineers reduce their S3 storage costs. The core problem is that S3 objects accumulate over time in expensive storage classes -- STANDARD costs $0.023/GB/month, but data that was hot a year ago is now cold, and nobody moves it to Glacier Instant Retrieval at $0.004/GB/month because auditing buckets manually across hundreds of prefixes does not scale. This is the same class of problem S3 Intelligent-Tiering solves at the platform level, but I wanted to build a tool that gives engineers full visibility and control over what gets moved and why, rather than a black-box policy. *(Customer Obsession -- engineers want to understand their costs, not just reduce them.)*

	- The application follows a four-phase workflow: **scan, score, execute, and rollback.**

	- **Scan** connects to a user's AWS account and crawls their S3 buckets using the `list_objects_v2` paginator. For each object, it checks `LastModified` against three thresholds: `_COLD_DAYS=90` for storage class transition candidates, `_STALE_DAYS=365` for potential deletion candidates, and `_MULTIPART_DAYS=7` for incomplete multipart uploads (found via `list_multipart_uploads` paginator). It also calls `get_bucket_lifecycle_configuration` to identify buckets missing lifecycle policies. Every object's storage class gets validated against a `StorageClass(str, Enum)` -- the scanner checks `StorageClass._value2member_map_`, logs a warning for unknown classes, and coerces to `None` rather than crashing. The scanner returns a 3-tuple `(recommendations, total_size_bytes, standard_size_bytes)` -- that separation of standard vs total was a critical bug fix I'll come back to. Each recommendation carries structured fields: `target_storage_class=StorageClass.GLACIER_IR` for transition recs, `upload_id=upload["UploadId"]` for multipart cleanup recs. *(Dive Deep -- the scanner doesn't just find "old objects," it classifies them against precise thresholds and validates every field.)*

	- **Score** runs each recommendation through a weighted risk model I designed. Five factors: reversibility 30%, data_loss_risk 25%, age_confidence 20%, size_impact 15%, access_confidence 10%. The scoring service uses a pricing matrix -- STANDARD $0.023, GLACIER_IR $0.004, DEEP_ARCHIVE $0.00099 -- and accounts for transition costs ($0.02/1000 requests for GLACIER_IR). The output is a 0-100 risk score. Risk >= 55, any delete action, or size >= 10GB triggers mandatory human approval. Risk < 30 with confidence >= 70 and no delete qualifies for safe automation. *(Are Right A Lot -- the thresholds are conservative by default because the cost of a false positive (moving hot data) is much higher than a false negative (leaving cold data in STANDARD for another cycle).)*

	- **Execute** applies changes in four progressive safety modes: `dry_run` (log only), `safe` (low-risk reversible only), `standard` (medium-risk), `full` (everything including deletes). Storage class transitions use `copy_object` with `MetadataDirective="COPY"` and `TaggingDirective="COPY"` -- this is the only way S3 supports in-place storage class changes, and the `COPY` directives preserve all user metadata and tags. Lifecycle policy additions use a merge strategy: the executor reads existing rules via `get_bucket_lifecycle_configuration`, merges new rules by stable ID to avoid duplicating or clobbering customer rules, then writes back with `put_bucket_lifecycle_configuration`. Multipart cleanup calls `abort_multipart_upload` using the dedicated `upload_id` field. The executor enforces none-safety guards -- if `target_storage_class` or `upload_id` is `None`, it fails fast with a clear error rather than making a malformed API call. Destructive actions (deletes) require dual controls: an env flag `ALLOW_DESTRUCTIVE_EXECUTION` AND the IAM permission `s3:DeleteObject`. Each action type maps to specific IAM permissions -- e.g., `CHANGE_STORAGE_CLASS` requires `["s3:GetObject", "s3:PutObject"]`. *(Insist on Highest Standards -- every mutation has explicit permission checks, mode guards, and none-safety validation before touching customer data.)*

	- **Rollback** captures pre-change state before every mutation. For storage class transitions, rollback calls `copy_object` again with the original storage class. For lifecycle policies, it restores the original rule set, or calls `delete_bucket_lifecycle` if no rules existed before. Delete actions are explicitly marked as not rollback-eligible -- the system never lies about what it can undo. *(Ownership -- I built rollback not because it was easy, but because anyone running this on production S3 buckets needs the ability to reverse a mistake within minutes.)*

	- On the architecture side, the backend is a Python FastAPI server using SQLite for persistence and boto3 for AWS operations. The frontend is a React/TypeScript SPA. I packaged everything as a Tauri v2 desktop application -- the Python backend gets frozen by PyInstaller into a standalone binary that Tauri spawns as a sidecar process. AWS credentials are stored in the OS keychain. The whole thing installs as a single DMG, MSI, or AppImage. *(Frugality -- zero infrastructure cost. SQLite, local credentials, no server.)*

	- The CI/CD pipeline runs tests across Python 3.11-3.13, builds Docker images with smoke tests, and produces cross-platform desktop installers. There are 245+ tests with 80%+ coverage enforced. It's been running on my own AWS account, saving roughly $50/month on S3 storage by automatically transitioning cold objects to Glacier IR -- an 83% reduction in per-GB cost on those objects. *(Deliver Results -- measurable dollar savings, not a theoretical tool.)*

- **Hardest Design Decisions:**

	- **Why progressive execution modes instead of a single "apply all" button?** I considered a simpler two-mode design (preview, execute). But I realized that a single execute mode forces a binary trust decision: either you trust the system with everything, or you trust it with nothing. Progressive modes let users build trust incrementally. They run `dry_run` to see what would happen, then `safe` to apply only low-risk reversible changes, then `standard` after seeing safe mode work correctly. This mirrors how S3 itself handles trust -- you don't give a new IAM role `s3:*` on day one; you start with read-only and escalate. The four modes cost more implementation complexity (mode eligibility checks on every action, a per-action permission mapping), but they dramatically reduce the risk of a user accidentally deleting production data on their first run. *(Earn Trust -- the system earns the user's trust through progressive disclosure, not a one-time EULA.)*

	- **Why CopyObject semantics for storage transitions?** S3 does not have a "change storage class" API. The only way to change an object's storage class in-place is to copy it to itself with a new `StorageClass` parameter. This means every storage transition is a PUT operation that costs money and temporarily doubles the object's storage footprint. I chose `MetadataDirective="COPY"` and `TaggingDirective="COPY"` explicitly because the default behavior for CopyObject is to strip metadata and tags -- silently losing customer metadata during a cost optimization would be catastrophic. This is a detail an S3 engineer would care deeply about: the API's default behavior is destructive, and you have to opt out of that destruction. *(Customer Obsession -- preserving customer metadata is non-negotiable, even when the API defaults to dropping it.)*

	- **Why pre-state capture architecture?** Before every mutation, the executor records exactly what the resource looked like. For storage transitions, it captures the original storage class. For lifecycle policies, it captures the full existing rule set. This means rollback is deterministic -- it restores to a known state, not a guessed one. The alternative was "undo by convention" (e.g., always transition back to STANDARD), but that fails when an object was already in GLACIER_IR and got transitioned to DEEP_ARCHIVE. Pre-state capture handles arbitrary starting states correctly. *(Think Big -- designing for correctness across all possible states, not just the common case.)*

- **Key Tradeoffs:**

	- **Synchronous scanning vs async.** The scanner makes synchronous boto3 calls, meaning a scan of a large bucket blocks the single Uvicorn worker and makes health checks unresponsive. I chose synchronous because aioboto3 adds complexity and is less battle-tested than boto3 for paginator patterns. The tradeoff is real -- on a bucket with millions of objects, the scan can take 30+ seconds during which the frontend shows a stale health status. If I were building this for a team at Amazon S3, async would be mandatory because blocking the server is unacceptable at any scale.

	- **SQLite vs Postgres for single-user.** SQLite ships embedded, requires zero configuration, and handles the workload (single writer, read-heavy, small data) perfectly. The constraint is single-writer -- I run Uvicorn with `--workers 1` and use a `threading.Lock` in RunStore. The RunStore class abstracts all DB operations, so migrating to Postgres means swapping `sqlite3` for `psycopg2`, adjusting a few SQL syntax differences, removing the lock, and increasing worker count. I chose the simpler option because operational complexity (running a Postgres instance) adds no value for a local desktop tool. *(Frugality -- don't pay for infrastructure you don't need.)*

	- **Heuristic weights vs ML-trained.** The scoring weights (30/25/20/15/10) are educated guesses, not trained on production data. An ML approach would track which recommendations users accept vs reject and train a model to predict acceptance probability. I chose heuristics because I don't have training data, the weights are transparent in every API response (users see exactly why something scored 72), and the approval/automation thresholds add a safety layer even if individual scores are slightly off. The architecture makes future ML integration trivial: swap the weight dictionary for a model prediction, keep the same threshold logic.

- **Difficult Bugs (Dive Deep + Highest Standards):**

	- **StorageClass enum corruption.** The original scanner had a subtle bug: for incomplete multipart upload recommendations, I was storing the `UploadId` string in the `storage_class` field of the Recommendation model. This worked in the scanner because the scorer parsed storage class as a string. But when the executor tried to use `storage_class` as an actual storage class for API calls, it would pass a UUID-like upload ID as a storage class parameter to `copy_object`, which would fail with a cryptic AWS error. The fix required three changes: (1) I created a proper `StorageClass(str, Enum)` with validation against `_value2member_map_` so unknown values get caught immediately, (2) I added dedicated fields -- `target_storage_class` for transition recs and `upload_id` for multipart cleanup recs, and (3) I added a Pydantic backward-compatibility validator so existing serialized data wouldn't break. This bug taught me that when you overload a field to mean two different things depending on context, you're creating a time bomb. *(Dive Deep -- the symptoms were cryptic AWS errors; the root cause was a type system violation three layers earlier.)*

	- **Lifecycle savings inflation.** The savings calculator was computing potential savings from ALL objects in a bucket, not just those in STANDARD storage. If a bucket had 100GB total but only 20GB in STANDARD (the rest already in GLACIER_IR), the calculator would report savings on 100GB -- inflating the projected savings by 5x. The fix was the 3-tuple return `(recommendations, total_size_bytes, standard_size_bytes)` and computing savings only against `standard_size_bytes`. This matters because inflated savings projections erode user trust -- if you promise $200/month savings and deliver $40, the user stops trusting your recommendations. *(Earn Trust -- accurate projections matter more than impressive-looking numbers. ### **21. Walk me through your Home Recommendation System.**

- **Answer:**

	- This is a full-stack, multi-stage recommendation platform for large-scale travel and accommodation. I built it to understand how production recommendation systems actually work end to end -- not just the model, but the data pipelines, real-time serving, infrastructure, and monitoring that make it production-grade. The tech stack is Java Spring Boot and Python on the backend, React on the frontend, PostgreSQL for storage, Kafka for streaming, and the ML pipeline uses TensorFlow, Pandas, and NumPy. The whole system is containerized with Docker, deployed on Kubernetes, and monitored with Prometheus and Grafana.

	- The core architecture is a **multi-stage recommendation pipeline** with three stages. **Stage 1: Candidate Generation** uses a Two-Tower neural network -- one tower encodes the user (preferences, history, behavioral signals) and the other tower encodes listings (features, location, price, amenities). Both towers produce embeddings in the same vector space, and candidate generation is an approximate nearest-neighbor lookup to retrieve the top few hundred candidates from the full inventory. I trained this on real-world datasets including RecSys Challenge 2021, Expedia Hotel Recommendations, and Airbnb New User Bookings -- over 1M transactional and behavioral records total. **Stage 2: Fine Ranking** takes those candidates and runs them through a Gradient Boosted Decision Tree (GBDT) model that incorporates both relevance signals and business rules -- blocked users, previously recommended items, and inventory constraints. This is where you go from "broadly relevant" to "specifically ranked for this user right now." **Stage 3: Re-Ranking** applies a Learn-to-Rank model that uses historical interaction feedback and real-time signals to produce the final ordering the user sees. The re-ranking stage is what lets the system adapt -- if a user just dismissed three beach properties, the re-ranker adjusts on the fly.

	- The **data layer** is built on ETL pipelines using Apache Spark and Airflow. These clean, transform, and engineer features from the raw datasets -- things like session duration aggregates, click-through rates per listing category, recency-weighted interaction scores. The feature store serves these engineered features at low latency for the real-time serving path. On the **serving side**, FastAPI handles live user requests, Kafka streaming processes interaction events in real time so the feature store stays current, and PostgreSQL stores user profiles, interaction history, and listing metadata. When a user requests recommendations, the system pulls their latest features from the store, runs the three-stage pipeline, and returns ranked results -- all within latency targets simulating Expedia-scale workloads.

	- The **infrastructure** is fully containerized with Docker and orchestrated on Kubernetes for scalability and reliability. I implemented a GitOps-based CI/CD workflow that automates testing, deployment, and rollback for all services. Prometheus and Grafana monitor latency, throughput, model drift, and system health in real time. The React frontend visualizes recommended items, user interactions, and model performance dashboards.

	- The biggest lesson from this project is that in a production recommendation system, the model is a small fraction of the total engineering. The pipeline orchestration, feature engineering, real-time serving infrastructure, monitoring, and deployment automation are where most of the complexity and most of the failure modes live. That's the perspective I'd bring to the S3 team -- if you're building intelligent storage features like Intelligent-Tiering or access pattern analysis, the algorithm matters, but the system around it is what determines whether it actually works reliably at exabyte scale.

- **Follow-Up Questions:**

	- **Q: Why a multi-stage pipeline instead of a single model?**

		- Scale and separation of concerns. A single model scoring every listing for every user on every request doesn't work at scale -- if you have millions of listings and millions of users, you can't run a heavy model on every possible pair. The multi-stage approach solves this by progressively narrowing. The Two-Tower model is cheap at inference time because it's just a vector similarity lookup -- you precompute listing embeddings and only compute the user embedding per request. That gets you from millions of candidates to a few hundred. Then the GBDT fine-ranker is more expensive per item but only runs on a few hundred candidates, not millions. The re-ranker is the most expensive per item but only touches the final top few dozen. Each stage does more work per item but on a smaller set, so the total compute stays manageable. It also lets you iterate independently -- I can swap the candidate generation model without touching the re-ranker, or add a new business rule to the GBDT stage without retraining the Two-Tower model.

	- **Q: Why Two-Tower for candidate generation specifically?**

		- Two-Tower is the industry standard for candidate generation for a reason. The key property is that the user tower and item tower are independent at inference time. You precompute all item embeddings offline and index them. At serving time, you only run the user tower (one forward pass) and then do approximate nearest-neighbor search against the precomputed item index. That makes candidate generation sub-millisecond even with millions of items. The alternative would be something like a cross-attention model that looks at user-item pairs jointly -- that gives better accuracy but requires running the model once per candidate, which is way too expensive for the candidate generation stage where you're scanning the full inventory. The tradeoff is that Two-Tower can't model fine-grained user-item interactions (like "this user specifically dislikes this amenity combination"), but that's exactly what the downstream GBDT and re-ranking stages handle.

	- **Q: Why GBDT for fine ranking instead of another neural network?**

		- GBDT handles heterogeneous features well -- you're mixing continuous signals (click-through rate, session duration) with categorical signals (listing type, location) and hard business constraints (blocked users, inventory limits). Neural networks can do this too, but GBDT is easier to incorporate hard business rules as features and to interpret which features are driving the ranking. In a real system, when a product manager asks "why did this listing rank above that one?" you can point to feature importances in the GBDT. With a deep neural ranker, that's harder to explain. The tradeoff is that GBDT doesn't capture sequential patterns or complex feature interactions as naturally as a neural model, but for a fine-ranking stage where you're already working with a curated candidate set, the interpretability and ease of incorporating business logic wins out.

	- **Q: How does the Learn-to-Rank re-ranking stage work?**

		- Learn-to-Rank optimizes for the ordering of results rather than individual relevance scores. The model is trained on historical interaction data -- which results in a ranked list did users actually click, engage with, or convert on. It learns pairwise preferences: "for this type of user, listing A should rank above listing B." The real-time component is what makes it powerful -- it incorporates signals from the current session. If a user has been dismissing high-rise apartments for the last ten minutes, the re-ranker picks up on that signal from the Kafka stream and deprioritizes similar listings without needing to retrain. The tradeoff versus just using the GBDT scores as final output is latency -- the re-ranker adds another inference step. But the improvement in user engagement justifies it because the re-ranker is the stage that makes recommendations feel adaptive rather than static.

	- **Q: Walk me through the data pipeline.**

		- The ETL pipeline starts with raw datasets -- user interaction logs, listing metadata, booking records. Apache Spark handles the heavy transformations: sessionizing click streams, computing rolling aggregates (7-day click-through rate, 30-day booking rate per category), joining user demographics with behavioral data, and encoding categorical features. Airflow orchestrates the DAGs so the pipeline runs on schedule and handles retries and alerts on failure. The engineered features land in both a batch store (PostgreSQL tables for model training) and a feature store optimized for low-latency reads during online serving. The feature store is the critical piece -- when a live request comes in, the serving layer needs the user's latest features in single-digit milliseconds, not the minutes it would take to recompute them. Kafka is the bridge between real-time and batch: interaction events stream through Kafka, get processed into feature updates, and land in the feature store so that the next recommendation request reflects recent behavior.

	- **Q: How did you handle the cold-start problem?**

		- The multi-stage architecture helps here. For a brand new user with no interaction history, the Two-Tower user embedding is initialized from their profile features only (demographics, stated preferences from onboarding). The embedding is sparse but functional -- it'll produce broadly reasonable candidates. The GBDT fine-ranker can still apply business rules and popularity signals even without personalization data. And the re-ranker starts learning from the user's in-session behavior through the Kafka stream almost immediately -- after just a few clicks, the re-ranking is already adapting. So cold-start degrades gracefully across all three stages rather than being a binary "works/doesn't work." The system is always returning something reasonable and getting more personalized with every interaction.

	- **Q: How do you monitor model quality in production?**

		- Prometheus collects metrics at every stage -- candidate generation latency, GBDT inference time, re-ranker throughput, end-to-end recommendation latency. Grafana dashboards visualize these in real time. For model quality specifically, I track model drift by comparing the distribution of prediction scores over time against the training distribution. If the GBDT's score distribution shifts significantly -- say the median confidence drops or the variance spikes -- that's a signal that the underlying data patterns have changed and the model needs retraining. I also track business metrics: click-through rate on recommended items, engagement depth (did the user view details or just scroll past), and conversion rate. A drop in any of these triggers an investigation -- is it a model issue, a data pipeline issue, or a product change that shifted user behavior?

	- **Q: What were the hardest infrastructure decisions?**

		- The biggest one was deciding where to draw the boundary between real-time and batch processing. You could try to make everything real-time -- retrain models on every new interaction, recompute all features in a streaming fashion -- but that's massively complex and expensive for marginal gains. I drew the line at: feature updates are real-time (via Kafka), model retraining is batch (scheduled Airflow DAGs), and the re-ranker bridges the gap by incorporating real-time signals without retraining. The second hard decision was Kubernetes resource allocation -- the candidate generation service has spiky load (lots of vector similarity lookups) while the GBDT service has steady load. I set up horizontal pod autoscaling with different policies for each service so the candidate generation pods can scale up quickly during traffic spikes without over-provisioning the GBDT pods.

	- **Q: How does this connect to what the S3 team does?**

		- The engineering principles transfer directly. S3's storage intelligence -- predicting access patterns, optimizing storage class placement, detecting anomalous usage -- is fundamentally the same class of problem: you have a pipeline that ingests signals, runs them through models, and produces a decision with production reliability requirements. The multi-stage architecture maps -- you'd have a fast initial classification (what access pattern does this object exhibit?), a more detailed optimization model (should this object transition to a colder storage class based on historical access?), and a final decision layer that incorporates business rules like minimum storage duration and transition costs. The infrastructure concerns are identical: low-latency serving, feature stores for real-time signal access, monitoring for model drift, and safe deployment so you can roll back a bad model without affecting customer data. The cold-start problem maps too -- a newly uploaded object has no access history, so you need deterministic rules as a baseline until the model has enough signal to recommend a storage class.

	- **Q: What would you do differently?**

		- Three things. First, A/B testing infrastructure from the start. I had no systematic way to run controlled experiments comparing one model against another with real traffic. In production, you need to be able to canary a new GBDT model to 5% of users and measure engagement before rolling it out fully. Second, I'd add an offline evaluation harness that replays historical interaction data through the pipeline to measure ranking quality (NDCG, MAP) before deploying a model change. I was relying on online metrics which are slower and noisier. Third, I'd invest more in the feature store's consistency guarantees -- there's a window where the Kafka consumer has processed an event but the feature store hasn't been updated yet, which means a request in that window gets stale features. In a travel platform that's acceptable, but in a storage management context like S3, that kind of staleness could matter for lifecycle transition decisions or access pattern classification.
