# AI and LLM Usage in Development
## AWS Cost Optimizer

This document describes how large language models were used during the development of AWS Cost Optimizer, the boundaries maintained between AI assistance and engineering judgment, and the productivity impact.

---

## 1. Philosophy

LLMs were used as a **productivity multiplier**, not a replacement for engineering decisions. The core architecture, risk model design, execution guardrails, and security posture were all human-designed. AI was leveraged for implementation velocity — turning well-defined specifications into working code faster than writing everything from scratch.

**Principle:** The engineer decides *what* to build and *why*. The LLM accelerates *how*.

---

## 2. Where AI Was Used

### 2.1 Code Generation (High Usage)

**Boilerplate and scaffolding:**
- Pydantic model definitions from a specification — field names, types, constraints, and validation rules were specified by the engineer; the LLM generated the model classes.
- FastAPI route handlers following the established pattern (request validation → service call → response construction).
- TypeScript interfaces mirroring the Python Pydantic models.
- Test scaffolding — given a service method and its expected behavior, generating pytest test functions with appropriate assertions.

**Implementation of well-specified logic:**
- The `RunStore` SQLite data access layer — schema was designed by the engineer, SQL operations and serialization logic were generated.
- Executor permission checking and mode eligibility logic — rules were specified as requirements, implementation was generated.
- Rollback handlers — behavior was defined (restore original storage class, restore original lifecycle rules), code was generated.

**What the engineer provided:**
- All API contracts (request/response shapes)
- Database schema design
- Scoring weight values and approval thresholds
- Execution mode definitions and eligibility rules
- Security requirements (destructive guard, permission validation)

### 2.2 Test Writing (High Usage)

**Test generation from specifications:**
- Unit tests were generated by describing the expected behavior of each method and the edge cases to cover.
- Boundary value tests (exact threshold values for risk scores, savings tiers, etc.) were specified by the engineer and generated by the LLM.
- Integration test workflows (scan → score → execute → rollback) were designed by the engineer and implemented with AI assistance.

**What the engineer ensured:**
- Test coverage targets (80%+ enforced)
- Correctness of assertions (manually reviewed)
- Test isolation (fixtures designed to prevent cross-test contamination)
- Edge cases that the LLM wouldn't naturally think of (empty inputs, double-write semantics, combined audit filters)

### 2.3 CI/CD Pipeline (Moderate Usage)

**Generated with guidance:**
- GitHub Actions workflow YAML for multi-version Python testing matrix
- Multi-platform Tauri build pipeline (macOS, Windows, Linux)
- Release pipeline with code signing and updater manifest generation

**Engineer-driven:**
- Pipeline architecture (which jobs depend on which)
- Security decisions (how to handle signing certificates, fallback behavior)
- Artifact structure and naming conventions

### 2.4 Documentation (High Usage)

**AI-assisted:**
- This documentation suite was generated from a comprehensive codebase analysis
- API documentation structure and formatting
- Test plan documentation

**Engineer-driven:**
- Architecture decision rationale (ADRs)
- Tradeoff analysis
- Future feature prioritization

### 2.5 Debugging and Problem Solving (Moderate Usage)

**Used for:**
- Diagnosing moto mocking issues (autouse fixture ordering, dual-binding patches)
- Resolving Tauri v2 configuration issues (CSP policies, sidecar paths)
- PyInstaller spec file configuration for cross-platform builds
- Understanding SQLite `:memory:` vs file-based behavior for test fixtures

**Not used for:**
- Root cause analysis was always done by the engineer reading error logs, understanding the stack, and forming hypotheses
- The LLM suggested solutions, but the engineer validated them against the codebase

## 3. Where AI Was NOT Used (Engineering Judgment)

### 3.1 Architecture Decisions
- Desktop vs web app — security and deployment tradeoff analysis was human reasoning
- Sidecar pattern — understanding PyInstaller limitations, Tauri process model, and startup timing
- SQLite vs PostgreSQL — workload analysis (single writer, read-heavy, small data) was domain expertise

### 3.2 Risk Model Design
- The five-factor weighted model, specific weights (30/25/20/15/10), and approval thresholds (risk >= 55, size >= 10 GB) were all human-designed based on understanding of S3 operations and their reversibility characteristics
- The scoring engine's behavior was specified first, then implemented with AI assistance

### 3.3 Security Posture
- Credential storage strategy (OS keychain, never plaintext)
- CSP configuration (restrict to localhost only)
- Destructive action gating (blocked by default, explicit opt-in)
- Progressive execution modes (dry_run → safe → standard → full)
- These are security design decisions that require understanding threat models, not code generation

### 3.4 Tradeoff Evaluation
- Single Uvicorn worker to avoid SQLite contention — understanding the interaction between process forking and file locking
- Synchronous vs async S3 calls — consciously accepting the limitation for v1
- Manual TypeScript mirroring vs code generation — weighing simplicity vs correctness guarantees

## 4. Productivity Impact

### 4.1 What Would Have Taken Longer Without AI

| Task | Estimated Time (Manual) | With AI | Savings |
|------|------------------------|---------|---------|
| Pydantic models (20+ models, 7 enums) | 3-4 hours | 30-45 min | ~80% |
| Test suite (245+ tests) | 16-20 hours | 4-5 hours | ~75% |
| CI/CD pipelines (3 workflows) | 4-6 hours | 1-2 hours | ~70% |
| API route handlers (8 endpoints) | 2-3 hours | 30-45 min | ~75% |
| RunStore data access layer | 3-4 hours | 1-1.5 hours | ~65% |
| TypeScript type definitions | 1-2 hours | 20-30 min | ~80% |

### 4.2 What Didn't Get Faster

| Task | Why |
|------|-----|
| Architecture design | Requires domain understanding and constraint analysis |
| Risk model tuning | Requires thinking through real-world S3 scenarios |
| Debugging sidecar startup | Required understanding process lifecycle and platform differences |
| CI signing configuration | Required understanding code signing, certificates, and platform-specific tooling |

### 4.3 Net Effect

AI assistance compressed what would have been weeks of implementation into days, while the engineering decisions (architecture, security, risk model, CI/CD strategy) still required the same amount of human thought. The project demonstrates that AI amplifies velocity on well-specified tasks but doesn't replace the judgment that defines those specifications.

## 5. Quality Assurance When Using AI

### 5.1 Review Process
- **Every generated file was read fully** before committing
- **Tests were run locally** before treating them as reliable
- **Generated code was tested against edge cases** the LLM might not anticipate
- **Security-sensitive code** (credential handling, execution guardrails) received extra scrutiny

### 5.2 Common AI Errors Caught
- Incorrect Pydantic v2 syntax (using v1 patterns like `class Config` instead of `model_config`)
- SQLite type mismatches (booleans as TEXT instead of INTEGER)
- Missing error handling for specific boto3 error codes
- Test assertions that were too loose (checking for truthiness instead of exact values)

### 5.3 Verification Strategy
- **Unit tests verify correctness** — if the generated code passes tests with exact expected values, it's correct
- **Integration tests verify contracts** — if the API returns the right shapes with the right status codes, the layers work together
- **Manual review verifies intent** — does the code do what I designed, not just what I described?

## 6. Lessons Learned

1. **Specify before generating.** The more precisely you define what you want (exact field names, types, constraints, edge case behavior), the better the output. Vague prompts produce vague code.

2. **AI is best at pattern repetition.** Once you establish a pattern (one test file, one route handler, one Pydantic model), the LLM can replicate it across the codebase with high accuracy.

3. **AI struggles with cross-cutting concerns.** Things like "the fixture needs to patch both import sites because of Python's module binding behavior" require understanding that the LLM can explain but rarely discovers on its own.

4. **Review the tests, not just the code.** AI-generated tests can have subtle flaws — testing the wrong thing, asserting against default values that happen to be correct, or missing critical edge cases. The test suite is your safety net; make sure the net is actually strong.

5. **Don't outsource architecture.** The most important decisions in this project — desktop vs web, weighted scoring vs simple classification, progressive execution modes, pre-change state capture — came from engineering reasoning about the problem domain. These are the decisions interviewers ask about, and they should reflect your thinking, not an LLM's.
